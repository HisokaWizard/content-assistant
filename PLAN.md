# План Проекта Content Assistant

## 1. Цель Продукта

Создать Telegram-бота, который помогает пользователю работать с контентом без необходимости читать все сообщения и смотреть все видео вручную.

Сервис должен:

- анализировать YouTube-видео по ссылке;
- собирать ежедневные сводки по публичным Telegram-каналам;
- поддерживать обычный диалог с пользователем на естественном языке;
- позволять каждому пользователю настраивать интересы, выбирать модель и управлять сессией.

Первая версия ориентирована на личное использование автором и тестовыми пользователями, но архитектура должна быть пригодна для дальнейшего запуска как публичного бота.

## 2. Основные Принципы Продукта

- Telegram-бот является только пользовательским интерфейсом.
- Backend отвечает за бизнес-логику, данные, лимиты, расписание и оркестрацию.
- Вся ИИ-логика должна находиться вне backend в виде агентов, тулов, скиллов и промптов, управляемых через статичные `md`-файлы на стороне `opencode`.
- Пользователь не должен знать ничего о внутренней агентной системе.
- У всех пользователей одинаковые функциональные возможности.
- Идентификация пользователя строится только по Telegram account.

## 3. Основные Пользовательские Сценарии

### 3.1 Универсальный Чат

Бот должен принимать сообщения пользователя в свободной форме.

Если сообщение является обычным текстом без команды и без YouTube-ссылки:

- отправить его в активную ИИ-сессию;
- вернуть ответ ассистента;
- учитывать размер контекста и дневные лимиты, зависящие от выбранной модели.

### 3.2 Анализ YouTube

Пользователь отправляет YouTube-ссылку в обычном сообщении или через специальную команду.

Ожидаемое поведение:

1. Обнаружить YouTube URL.
2. Запустить задачу на обработку.
3. Показывать пользователю прогресс.
4. Попытаться получить транскрипт бесплатным способом.
5. Если транскрипт недоступен, попытаться выполнить speech-to-text.
6. Если видео не на русском языке, перевести результат на русский.
7. Передать нормализованный материал ИИ-агенту для структурированного ответа.

Начальный ответ должен включать:

- краткое summary;
- ключевые тезисы;
- рекомендацию, стоит ли смотреть видео;
- оценку по пользовательским критериям просмотра.

Аудиоответ не входит в MVP.

### 3.3 Сводки По Telegram-Каналам

Пользователь добавляет публичный Telegram-канал по ссылке или `@username`.

Ожидаемое поведение:

1. Проверить доступность канала.
2. Классифицировать канал.
3. Сохранить его в общем реестре каналов.
4. Привязать канал к подпискам пользователя.
5. Включить канал в ежедневную генерацию сводок.

Доставка сводки:

- автоматически, один раз в день;
- с группировкой по темам:
  - технологии;
  - крипта;
  - экономика;
  - политика.

Содержимое сводки должно приоритизировать:

- повторяющиеся темы в разных каналах;
- прогнозы;
- ключевые тезисы;
- факты;
- идеи;
- дропы и запуски;
- прорывные и важные события;
- сигналы, способные изменить дальнейшую повестку.

В MVP обрабатываются только текстовые сообщения.

## 4. Границы MVP

### Входит В MVP

- Telegram-бот с обычным чатом и командами;
- backend-сервис на `Node.js + TypeScript`;
- внешний `opencode`-сервис для выполнения ИИ-задач;
- анализ YouTube-ссылок;
- ежедневные сводки по Telegram-каналам;
- пользовательские интересы;
- выбор модели для каждого пользователя;
- одна активная сессия на пользователя;
- выбор и удаление старых сессий из истории;
- отслеживание заполнения контекстного окна с предупреждением на 75%;
- дневные лимиты, зависящие от модели;
- блокировка всех ИИ-действий при исчерпании дневного лимита;
- уведомление пользователя о необходимости сменить модель после достижения лимита.

### Не Входит В MVP

- аудиоформат summary;
- приватные Telegram-каналы;
- обработка изображений, видео, голосовых, файлов и опросов из каналов;
- несколько одновременно активных сессий;
- веб-интерфейс;
- админ-панель;
- роли и разграничение прав доступа.

## 5. Архитектура Верхнего Уровня

### 5.1 Telegram-Бот

Зоны ответственности:

- принимать команды и сообщения в свободной форме;
- обнаруживать YouTube-ссылки;
- показывать прогресс выполнения задач;
- отображать ответы ИИ;
- отправлять ежедневные сводки;
- предоставлять пользовательские команды управления.

Бот должен оставаться тонким клиентским слоем и не содержать внутри себя бизнес-логику.

### 5.2 Backend-Сервис

Рекомендуемый стек:

- `Node.js`;
- `TypeScript`;
- `PostgreSQL`;
- ORM или query builder, который будет выбран позже;
- scheduler для ежедневных задач;
- очередь для долгих операций, если понадобится.

Зоны ответственности:

- управление пользователями;
- управление интересами;
- управление каналами и подписками;
- управление сессиями и историей сессий;
- получение списка доступных моделей с ИИ-стороны;
- применение лимитов;
- отслеживание заполнения контекста;
- выполнение scheduled-задач для сводок;
- хранение метаданных задач и результатов;
- координация вызовов к `opencode`.

### 5.3 AI / Opencode Слой

Зоны ответственности:

- основной диалоговый агент;
- сабагенты для анализа YouTube и генерации дайджестов;
- tools и skills, описанные в статичных `md`-файлах;
- суммаризация транскриптов;
- перевод;
- классификация;
- генерация структурированных сводок;
- обработка диалога с учетом сессии.

Backend должен воспринимать этот слой как движок выполнения, а не как место хранения постоянных продуктовых данных.

## 6. Модель Данных

Рекомендуемая база данных: `PostgreSQL`.

Причины:

- предметная область естественно укладывается в реляционную модель;
- нужен единый реестр каналов без дублей;
- удобно реализуются связи подписок, сессий и истории;
- SQL лучше подходит для надежной работы с расписанием, отчетностью и ростом проекта;
- решение хорошо масштабируется для будущего публичного запуска.

## 7. Требования К Файловой Структуре

Для каждого пользователя на сервере должна существовать отдельная директория.

Назначение:

- привязка к нативной механике сессий `opencode`;
- сохранение непрерывности пользовательской сессии;
- поддержка выбора и удаления старых сессий;
- изоляция пользовательского рабочего состояния ИИ.

Пример концепции:

- `/data/users/<telegram_id>/`

Возможная вложенная структура:

- `sessions/`
- `artifacts/`
- `logs/`
- `temp/`

Точная структура может быть окончательно определена на этапе реализации.

## 8. Команды Бота

Финальный список команд может уточняться, но в MVP должны быть команды следующих категорий.

### Пользовательские Настройки

- `/interest <text>`: добавить или обновить интересы пользователя
- `/criteria <text>`: задать критерии оценки, стоит ли смотреть видео
- `/model`: показать доступные модели
- `/use_model <model>`: переключить активную модель

### Сессии

- `/session_current`: показать текущую активную сессию
- `/session_history`: показать историю прошлых сессий
- `/session_select <id>`: переключиться на старую сессию
- `/session_delete <id>`: удалить сессию
- `/session_clear`: очистить контекст текущей сессии

### YouTube

- `/analyze <youtube_url>`: проанализировать YouTube-видео

Обычные сообщения с YouTube URL должны автоматически запускать тот же сценарий.

### Telegram-Каналы

- `/add_channel <url_or_username>`: добавить публичный канал
- `/channels`: показать список подключенных каналов
- `/remove_channel <id>`: удалить канал из подписок пользователя
- `/digest_now`: запросить актуальную сводку вручную

## 9. Правила Лимитов И Контекста

### Дневные Лимиты

- лимиты зависят от выбранной модели;
- все действия, связанные с ИИ, расходуют единый дневной бюджет;
- при исчерпании лимита все ИИ-действия блокируются;
- бот сообщает пользователю о достижении лимита и предлагает сменить модель.

### Заполнение Контекста

- для каждой активной сессии нужно отслеживать приблизительное заполнение контекстного окна;
- при достижении 75% бот предупреждает пользователя;
- в MVP автоматическая очистка не выполняется;
- пользователь сам решает, продолжать ли работу, очищать контекст или переключаться на другую сессию.

## 10. Стратегия Обработки YouTube

Целевой порядок работы:

1. Сначала попытаться получить бесплатный транскрипт.
2. Если это невозможно, запустить speech-to-text.
3. Определить язык исходного материала.
4. Если нужно, перевести на русский язык.
5. Передать нормализованный материал ИИ-агенту для структурированного ответа.

Важное техническое замечание:

Получение YouTube-транскриптов и fallback через speech-to-text являются зонами технического риска, поэтому их нужно изолировать за отдельным adapter-интерфейсом.

## 11. Стратегия Генерации Telegram-Сводок

Pipeline ежедневной сводки:

1. Загрузить все активные подписки пользователя.
2. Прочитать новые текстовые сообщения из публичных каналов.
3. Нормализовать и дедуплицировать контент.
4. Скластеризовать сообщения по темам.
5. Выделить повторяющиеся темы между каналами.
6. Извлечь наиболее важные элементы:
   - прогнозы;
   - факты;
   - идеи;
   - запуски;
   - прорывы;
   - важные сдвиги повестки.
7. Сгенерировать сжатую структурированную сводку через ИИ-агента.
8. Автоматически отправить ее пользователю.

## 12. Риски И Ограничения

### Технические Риски

- бесплатные способы получения транскриптов YouTube могут быть нестабильны;
- speech-to-text может быть медленным и дорогим;
- чтение Telegram-каналов может потребовать клиентскую интеграцию, а не только bot API;
- оценка заполнения контекста может отличаться между моделями и провайдерами;
- хранение пользовательских сессий на диске должно быть спроектировано аккуратно, чтобы избежать повреждения или потери состояния.

### Продуктовые Риски

- ежедневные сводки могут стать шумными без сильной приоритизации;
- UX переключения моделей может стать запутанным, если метаданные провайдеров нестабильны;
- ограничения бесплатных моделей могут заметно снижать качество ответов и ухудшать пользовательский опыт.

## 13. План Реализации

### Этап 0. Основа

- инициализировать структуру репозитория или сервисов;
- определить границы компонентов;
- поднять backend-проект на `TypeScript`;
- настроить базу данных и миграции;
- подготовить каркас Telegram-бота;
- подготовить контракт интеграции с `opencode`.

### Этап 1. Пользователь И Сессии

- реализовать создание пользователя по Telegram identity;
- реализовать хранение настроек;
- реализовать управление интересами;
- реализовать модель одной активной сессии;
- реализовать выбор и удаление сессий из истории;
- создать файловые директории пользователей.

### Этап 2. Базовая Оркестрация Чата

- маршрутизировать сообщения из бота в backend и AI-слой;
- получать список доступных моделей с AI-стороны;
- реализовать переключение моделей;
- хранить историю сообщений и usage;
- реализовать применение дневных лимитов;
- реализовать предупреждение о заполнении контекста на 75%.

### Этап 3. YouTube MVP

- обнаруживать YouTube URL;
- реализовать transcript adapter;
- реализовать fallback через speech-to-text;
- реализовать fallback через перевод;
- реализовать структурированный ИИ-ответ;
- показывать пользователю прогресс выполнения;
- добавить явную команду `/analyze`.

### Этап 4. Telegram-Каналы MVP

- реализовать добавление, удаление и просмотр каналов;
- реализовать классификацию каналов;
- хранить подписки через общий реестр каналов;
- собирать текстовые посты из публичных каналов;
- генерировать автоматические ежедневные сводки;
- добавить ручной запуск через `/digest_now`.

### Этап 5. Стабилизация

- улучшить обработку ошибок;
- добавить ретраи и видимость состояния задач;
- уточнить промпты, скиллы и контракты тулов;
- оптимизировать контроль токенов и расходов;
- подготовить документацию по деплою и эксплуатации.

## 14. Критерии Готовности MVP

MVP считается готовым, когда выполняются все следующие условия:

- пользователь может открыть Telegram-бота и общаться с ним на естественном языке;
- пользователь может отправить YouTube-ссылку и получить полезное русскоязычное summary с тезисами и рекомендацией, смотреть видео или нет;
- пользователь может задать интересы и критерии оценки через команды;
- пользователь может добавить публичные Telegram-каналы и получать автоматическую ежедневную сводку;
- пользователь может переключать модель из списка доступных;
- пользователь может просматривать историю сессий, выбирать сессию, очищать текущую и удалять старые;
- backend надежно хранит данные пользователя, сессии, подписки и состояние использования ИИ;
- выполнение ИИ-логики изолировано в слое `opencode` через определения агентов и тулов.

## 15. Следующий Артефакт Планирования

До начала кодинга следующим документом нужно определить:

- точную структуру репозитория;
- API-контракты между bot, backend и `opencode`;
- точную схему базы данных;
- жизненный цикл задач и статусы очередей;
- формат ответов команд;
- модель деплоя для локальной и production-среды.
